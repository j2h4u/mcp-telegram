# Phase 999.1: Code Review Report

**Reviewed:** 2026-04-24
**Depth:** standard
**Files Reviewed:** 15
**Status:** issues_found

## Summary

Phase 999.1 introduces the own-message activity archive: a new `activity_comments` table (schema v14), a background `activity_sync` worker using `SearchRequest`/`iter_messages`, a `_get_my_recent_activity` daemon handler, and the `GetMyRecentActivity` MCP tool. A `fragment` branch is added to `_list_messages` for point-fetching context around specific message IDs without full sync. Overall the design is sound; the daemon handler, sync loop, and tool formatter are clean. However one critical Python 3 syntax error makes the entire daemon non-importable, several logic gaps exist in the incremental sync path, and test coverage has two audit gaps.

---

## Critical Issues

### CR-01: Python 3 syntax error — daemon is entirely non-importable

**File:** `src/mcp_telegram/daemon_api.py:843`

**Issue:** `except ValueError, KeyError:` is Python 2 syntax. Python 3 requires parentheses: `except (ValueError, KeyError):`. The comma form is a `SyntaxError` in Python 3 that prevents `daemon_api` from being compiled at all. Every import of this module — including `daemon.py`, all MCP tools via `daemon_client.py`, and the entire test suite — fails at import time.

```
$ python3 -m py_compile src/mcp_telegram/daemon_api.py
  File "src/mcp_telegram/daemon_api.py", line 843
    except ValueError, KeyError:
           ^^^^^^^^^^^^^^^^^^^^
SyntaxError: multiple exception types must be parenthesized
```

**Fix:**
```python
# line 843 — before
except ValueError, KeyError:
# after
except (ValueError, KeyError):
```

## Warnings

### WR-01: Incremental sync — `iter_messages(entity=None)` may not be valid for global search

**File:** `src/mcp_telegram/activity_sync.py:242`

**Issue:** `_run_incremental` calls `client.iter_messages(entity=None, from_user="me")`. In Telethon, `iter_messages` with `entity=None` typically triggers `get_messages` in "global" mode which is not well-defined for all account types and historically raises `ValueError: Cannot find any entity corresponding to "me"` when the session is not fully resolved. The backfill path uses the validated `SearchRequest(peer=InputPeerEmpty(), from_id=InputPeerSelf())` pattern which is the correct Telegram API surface for global own-message search. The incremental path silently diverges from this contract. If `iter_messages(entity=None)` does not work in production (e.g., raises `FloodWaitError` or `TypeError`), the outer exception handler in `run_activity_sync_loop` swallows it and logs a generic warning, meaning failures are invisible.

**Fix:** Use the same `SearchRequest` with `min_id=max_message_id_seen` instead of `iter_messages(entity=None)`, or at minimum add an explicit guard around the `entity=None` call with a test-verified integration note.

```python
# Replace _run_incremental's API call with:
result = await client(SearchRequest(
    peer=InputPeerEmpty(),
    q="",
    filter=InputMessagesFilterEmpty(),
    min_date=None,
    max_date=None,
    offset_id=0,
    add_offset=0,
    limit=100,
    max_id=0,
    min_id=max_message_id_seen,   # only newer messages
    hash=0,
    from_id=InputPeerSelf(),
))
```

### WR-02: Incremental sync — break condition uses `sent_at` instead of `message_id`

**File:** `src/mcp_telegram/activity_sync.py:246-247`

**Issue:** The incremental loop stops when `sent_at <= max_sent_at`:
```python
sent_at = int(msg.date.timestamp()) if getattr(msg, "date", None) else 0
if sent_at <= max_sent_at:
    break
```
`max_sent_at` is `MAX(sent_at)` from `activity_comments`. Two problems:
1. `sent_at` can be identical for multiple messages sent in the same second. The break will fire too early, silently dropping any sibling messages that share the same timestamp as the highest-stored message.
2. Telegram `iter_messages` returns results newest-first by default. If the user sent two messages at the same second and only one is already stored, the second will be skipped. The correct anchor is `MAX(message_id)`, not `MAX(sent_at)`.

**Fix:**
```python
# In _run_incremental, change the anchor query and break condition:
row = conn.execute("SELECT MAX(message_id) FROM activity_comments").fetchone()
max_msg_id = int(row[0]) if row and row[0] is not None else 0

if max_msg_id == 0:
    return

async for msg in client.iter_messages(entity=None, from_user="me"):
    if shutdown_event.is_set():
        break
    if msg.id <= max_msg_id:
        break
    r = _message_to_row(msg)
    ...
```

### WR-03: `_fetch_fragment_context` — bare `except` swallows all errors silently

**File:** `src/mcp_telegram/daemon_api.py:2541-2546`

**Issue:**
```python
except Exception:
    logger.warning(
        "fragment_fetch_failed dialog_id=%s anchor=%s",
        dialog_id, anchor_message_id, exc_info=True,
    )
    return
```
When `get_input_entity` or `get_messages` fails, `_fetch_fragment_context` returns `None` silently. The calling branch then immediately calls `_list_messages_context_window` which queries the now-empty messages table for `dialog_id`. If no rows were inserted (because the fetch failed), the context window returns zero messages and a `coverage='fragment'` response with an empty list — no error is propagated to the caller. The LLM receives an empty response indistinguishable from "no messages in that window" rather than "fetch failed."

**Fix:** Return a boolean from `_fetch_fragment_context` indicating success/failure, and propagate an error response from `_list_messages` when the fragment fetch fails:
```python
async def _fetch_fragment_context(self, dialog_id: int, anchor_message_id: int) -> bool:
    ...
    except Exception:
        logger.warning("fragment_fetch_failed ...", exc_info=True)
        return False
    ...
    return True

# In _list_messages, fragment branch:
if current_status in (None, "not_synced", "fragment"):
    ok = await self._fetch_fragment_context(dialog_id, context_message_id)
    if not ok:
        return {"ok": False, "error": "fragment_fetch_failed",
                "message": "Could not fetch messages from Telegram."}
    ...
```

### WR-04: `_get_my_recent_activity` — no row_factory set, positional indexing is fragile

**File:** `src/mcp_telegram/daemon_api.py:2621-2657`

**Issue:** The `_get_my_recent_activity` query accesses columns by positional index (`r[0]` through `r[6]`). The `DaemonAPIServer.__init__` sets `conn.row_factory = sqlite3.Row` globally, so `r[0]` etc. work correctly today. However the comment at line 710 documents that `conn.row_factory = sqlite3.Row` is set by `__init__` — the `_get_my_recent_activity` handler is the only new handler in this phase that does not convert its rows through a dataclass or named-key extraction, making it silently position-dependent.

More critically, the query joins `entities e` but the column order mixes `ac.*` and `e.name AS dialog_name`. If the SELECT column order is ever changed (e.g., to add `last_synced_at`), the positional mapping silently produces wrong data.

**Fix:** Use `sqlite3.Row` named access consistently:
```python
comments = [
    {
        "dialog_id": r["dialog_id"],
        "message_id": r["message_id"],
        "sent_at": r["sent_at"],
        "text": r["text"],
        "reactions": r["reactions"],
        "reply_count": r["reply_count"],
        "dialog_name": r["dialog_name"] if r["dialog_name"] else str(r["dialog_id"]),
    }
    for r in rows
]
```

### WR-05: `test_no_telethon_imports_in_tools` does not cover `activity.py`

**File:** `tests/test_tool_routing.py:523`

**Issue:** The architectural invariant test checks a fixed list of filenames:
```python
for filename in ["discovery.py", "reading.py", "_base.py", "sync.py", "user_info.py", "unread.py"]:
```
The new `activity.py` is not in this list. If someone later adds a Telethon import to `activity.py`, the guard will not catch it.

**Fix:**
```python
tools_dir = pathlib.Path(__file__).parent.parent / "src" / "mcp_telegram" / "tools"
for filepath in tools_dir.glob("*.py"):
    if filepath.name.startswith("__"):
        continue
    content = filepath.read_text()
    assert "from telethon" not in content, f"{filepath.name} imports telethon"
    assert "import telethon" not in content, f"{filepath.name} imports telethon"
```
(This mirrors the pattern already used in `test_no_connected_client_in_tools`.)

### IN-02: `_set_state` silently no-ops when the key row is missing

**File:** `src/mcp_telegram/activity_sync.py:56-61`

**Issue:** `_set_state` uses `UPDATE ... WHERE key=?`. If the key row does not exist in `activity_sync_state` (e.g., a future migration error leaves the seed `INSERT OR IGNORE` incomplete), the UPDATE matches zero rows and silently succeeds. The `backfill_complete` state never gets written, so `_run_incremental` never runs and `_run_backfill` loops indefinitely on every restart.

**Fix:** Use `INSERT OR REPLACE` or check `rowcount`:
```python
def _set_state(conn: sqlite3.Connection, key: str, value: str | None) -> None:
    with conn:
        conn.execute(
            "INSERT OR REPLACE INTO activity_sync_state (key, value) VALUES (?, ?)",
            (key, value),
        )
```

### IN-03: `_message_to_row` — missing `text` when `msg.message` is empty string

**File:** `src/mcp_telegram/activity_sync.py:93`

**Issue:** `getattr(msg, "message", None)` stores empty string `""` as `text` for service messages or messages with only media. This is consistent with the main `messages` table, but the tool formatter in `activity.py` does `(comment.get("text") or "").replace("\n", " ")`, which silently replaces `None` and `""` the same way. No functional bug, but `""` vs `None` is semantically ambiguous — `None` means "no text field", `""` means "empty text field". Worth normalizing to `None or the actual text` for consistency.

**Fix:** Normalize to `None` when empty:
```python
text = getattr(msg, "message", None) or None
```
