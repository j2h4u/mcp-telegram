  <files>src/mcp_telegram/tools.py</files>
  <action>
Update the non-General topic fetch path in `list_messages` so `TOPIC_ID_INVALID` is treated as a recoverable topic-state mismatch before it becomes a terminal user-visible error.

Implementation targets:
- Detect the thread-fetch failure branch where a resolved topic is using `reply_to=top_message_id`.
- On the first `TOPIC_ID_INVALID`-style failure only, call `_refresh_topic_by_id(...)` for the resolved topic ID.
- If the refreshed row is deleted, return deleted-topic text.
- If the refreshed row is still active and its `top_message_id` changed, retry the fetch once with the new anchor and persist that metadata in cache.
- If the refreshed row is active but still cannot be fetched, return inaccessible-topic text that preserves the topic context and does not imply success.

Keep this path bounded to one refresh + one retry. Do not add an unbounded loop, and do not broaden the fallback to dialog-wide history.
  </action>

    <automated>uv run pytest tests/test_tools.py -k "stale_top_message_id or deleted_after_refresh or inaccessible_after_refresh" -v</automated>
