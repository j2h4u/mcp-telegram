# Milestone v1.2: MCP Surface Research

**Status:** ✅ SHIPPED 2026-03-13
**Phases:** 10-13
**Total Plans:** 12

## Overview

`v1.2` was a research-only milestone. It anchored the future MCP surface refactor in primary-source
guidance plus the actual `mcp-telegram` brownfield runtime, then turned that evidence into a
decision-ready recommendation and implementation brief for the next coding milestone.

## Phases

### Phase 10: Evidence Base & Audit Frame

**Goal**: Maintainer has a source-ranked evidence base and audit frame for evaluating the current
MCP surface without drifting into a vague literature review.
**Depends on**: Nothing (first phase of `v1.2`; audits the shipped `v1.1` surface)
**Plans**: 3 plans

Plans:

- [x] 10-01: Capture the retained-source evidence hierarchy and reflected tool inventory
- [x] 10-02: Freeze the brownfield baseline, workflow burden, and preserved invariants
- [x] 10-03: Define the reusable audit rubric and later-phase handoff rules

**Details:**
Established the evidence hierarchy, locked the seven-tool reflected runtime baseline, and created a
project-specific audit rubric reused by Phases 11-13.

### Phase 11: Current Surface Comparative Audit

**Goal**: Maintainer can review a grounded comparative audit of the current MCP surface from the
LLM-facing perspective.
**Depends on**: Phase 10
**Plans**: 3 plans

Plans:

- [x] 11-01: Audit each public tool against the Phase 10 rubric
- [x] 11-02: Audit end-to-end workflows, recovery behavior, and helper-step burden
- [x] 11-03: Synthesize the current-state findings into one decision-ready comparative audit

**Details:**
Converted the retained evidence into a tool-level and workflow-level audit, including preserved
strengths, leak taxonomy, recovery boundaries, and a single decision-ready synthesis.

### Phase 12: Redesign Options & Pareto Recommendation

**Goal**: Maintainer can compare redesign paths and review one evidence-backed Pareto
recommendation for the next milestone.
**Depends on**: Phase 11
**Plans**: 3 plans

Plans:

- [x] 12-01: Define the comparison frame and invariant-aware contract-delta rules
- [x] 12-02: Map minimal, medium, and maximal redesign options across the current surface
- [x] 12-03: Select and justify the Medium-path Pareto recommendation

**Details:**
Compared three redesign ranges against the same baseline and invariants, then selected the Medium
path as the smallest safe change set likely to remove most model burden.

### Phase 13: Implementation Sequencing & Decision Memo

**Goal**: Maintainer has a decision-ready memo that turns the research into a sequenced,
validate-able implementation brief for the follow-up milestone.
**Depends on**: Phase 12
**Plans**: 3 plans

Plans:

**Details:**
Turned the recommendation into a concrete implementation memo with ordered sequencing, runtime

## Milestone Summary

**Decimal Phases:**

- None

**Key Decisions:**

- Use official MCP and Anthropic guidance as normative external inputs, and reflection/code/tests
  as brownfield authority.
- Freeze all redesign analysis against the reflected seven-tool runtime rather than inherited notes.
- Choose the Medium path as the next milestone because it removes substantial agent burden with
  the smallest safe change set.
- Treat reflected-schema checks plus restarted-runtime freshness as mandatory acceptance gates once
  public tool schemas move.

**Issues Resolved:**

- Replaced ad hoc design discussion with a retained-source evidence hierarchy and reusable audit frame.
- Unified tool-level and workflow-level findings into one decision-ready current-surface audit.
- Converted the research output into a sequenced implementation memo instead of disconnected notes.

**Issues Deferred:**

- Phase 10-13 validation artifacts remain partial and may need follow-up if Nyquist closure is required later.
- Large-forum validation and some v1.1 cleanup remain backlog inputs rather than `v1.2` scope.

**Technical Debt Incurred:**

- The primary Phase 13 memo still relies on supporting artifacts for the clearest provenance back to
  the Phase 10 evidence hierarchy and audit rubric.
- Runtime freshness remains an operational risk until the follow-on implementation milestone executes
  rebuild/restart verification against the live container.
