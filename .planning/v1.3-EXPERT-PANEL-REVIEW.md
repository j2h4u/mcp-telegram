## System Architect

**Assessment:** The layering is sound — `server.py` -> `tools.py` -> `capabilities.py` -> `cache/resolver/formatter/pagination` is a clean progression. The v1.3 capability extraction was well-executed: tools are thin adapters, shared behavior lives in one place.

**Risks:** `tools.py` (1781 LOC) and `capabilities.py` (1899 LOC) are both large and still carry duplicate constants. The string-based tool registry is fragile.

**Recommendation:** Extract shared constants to avoid drift. The registry pattern in `server.py` relies on class introspection + string matching — a registration decorator would be safer.

**Open question:** Is the `tools.py` <-> `capabilities.py` split the right seam long-term, or should tool runners become thin wrappers over a single orchestration layer?

### Findings

- [ ] **A-1. String-Based Tool Registry & Runtime Dispatch** (Critical)
  - `server.py:41`, `tools.py:149-203`
  - Tool argument instantiation via `sys.modules[__name__].__dict__[tool.name](**kwargs)` creates brittle binding. If a tool class is renamed or registration is forgotten, error only surfaces at runtime. Tool discovery (introspect ToolArgs subclasses) and runner registration (`@singledispatch` decorators) are two-phase setup that can fall out of sync.
  - **Fix:** Replace with explicit tool registry dict mapping tool name -> (ToolArgs class, async runner). Add startup verification that all discovered tools have registered runners.

- [x] **A-2. Constants Defined in Two Locations** (DRY violation)
  - `tools.py:40-44`, `capabilities.py:31-34`
  - Four constants duplicated: `FORUM_TOPICS_PAGE_SIZE`, `TOPIC_METADATA_TTL_SECONDS`, `GENERAL_TOPIC_ID`, `GENERAL_TOPIC_TITLE`. Any change must be made in two places.
  - **Fix:** Keep canonical definitions in `capabilities.py`, import in `tools.py`. Or extract to shared `constants.py`.

- [x] **A-3. Parameter Sprawl in Capability Functions** (Readability, correctness risk)
  - `capabilities.py:800-812` and 10+ similar functions
  - `resolve_exact_topic_target` has 11 parameters; three (`exact_topic_id`, `exact_topic_name`, `exact_topic_metadata`) are alternative ways to resolve the same entity. Callers must know which combinations are valid.
  - **Fix:** Extract `TopicResolutionContext` dataclass to group optional resolution hints, reducing params to ~7.

- [ ] **A-4. Tight Coupling: tools.py Calls Capability Functions Directly**
  - Changes to capability function signatures require ripple edits in tools.py. No abstraction boundary; `capabilities.py` is a helper module rather than a true layer.
  - **Fix:** Define a Capability Protocol that tools depend on. Allows future swapping of implementations without tool changes.

- [ ] **A-5. Cache Lifecycle Management**
  - `tools.py:207-212`, `cache.py:209-215`
  - EntityCache created via `@functools_cache`, opened once. ReactionMetadataCache and TopicMetadataCache initialized on-demand from shared connection. No explicit cleanup — connections may not close on early exception.
  - **Fix:** Create `CacheManager` class owning all three caches with context manager lifecycle.

- [ ] **A-6. No Explicit Tool Registration Audit Trail**
  - `tools.py:150-153` (singledispatch), decorators at lines 1034, 1113, 1255, 1379, 1469, 1526, 1683
  - Registration is implicit via scattered decorators. A reviewer must manually count to verify all tools are registered. No startup verification.
  - **Fix:** Add startup check comparing expected tools set vs registered set.

## Clean Code Specialist

**Assessment:** Code is mostly clean. Pydantic models are well-used. The biggest smells are constant duplication, repetitive telemetry boilerplate, and weak type hints on formatter callbacks.

**Risks:** Copy-paste telemetry code across all 7 tool runners (~560 LOC of boilerplate). Parameter sprawl in capabilities.py. Untyped `object` and `list` in formatter signatures leak abstraction boundaries.

**Open question:** Are the `getattr` chains in `formatter.py` defensive against real Telethon quirks, or overly cautious?

### Findings

- [x] **C-1. Telemetry Boilerplate Duplication** (Major DRY violation)
  - `tools.py` — all 8 tool runners, ~560 total LOC
  - Every tool repeats ~70 lines of telemetry setup/teardown: logger.info, t0 = time.monotonic(), error_type tracking, try/except/finally with TelemetryEvent recording. 8 tools x 70 lines. Bug fixes require 8 edits.
  - **Fix:** Extract `@track_telemetry("ToolName")` decorator factory. Reduces ~560 LOC to ~50. Each tool runner becomes pure business logic.

- [ ] **C-2. Weak Type Hints Throughout** (Low IDE support, clarity issues)
  - `tools.py:64` (`entity: t.Any`), `capabilities.py:565, 801` (`client: object`), `formatter.py:10, 157, 206` (`msg: object`), `tools.py:23` (imports available but unused in signatures)
  - IDE autocomplete doesn't work; Mypy can't validate; refactoring is fragile.
  - **Fix:** Replace `object` with specific Telethon types (`TelegramClient`, `Message`, etc.) or define Protocol types for the message interface.

- [x] **C-3. Inconsistent Error Response Construction**
  - `tools.py:1074, 1481, 1423`
  - Three different patterns for building TextContent responses across tools. No consistent factory.
  - **Fix:** Add `success_text_response(text)` and `error_response(failure)` builder functions.

- [x] **C-4. Magic Numbers Without Explanation**
  - `formatter.py:6` (`SESSION_BREAK_MINUTES = 60`), `tools.py:40` (`REACTION_NAMES_THRESHOLD = 15`)
  - No comments explaining rationale. These are policy decisions.
  - **Fix:** Add comments explaining *why* these values were chosen.

- [ ] **C-5. Enum-Like Constants Without Actual Enum**
  - `capabilities.py:35-37`
  - `HISTORY_NAVIGATION_NEWEST/OLDEST` as string constants + `Literal` type + manual validation at `pagination.py:78`. Redundant three-way definition.
  - **Fix:** Use `StrEnum` for type safety and single definition.

- [ ] **C-6. Long Methods with Multiple Responsibilities**
  - `tools.py:1255-1350` (list_messages runner, ~95 LOC)
  - One async function handles: logging, timer, dialog resolution, forum topic loading, sender resolution, message fetching, reply/reaction building, formatting, response construction, telemetry.
  - **Fix:** Decompose into `_execute_list_messages_capability()` and `_format_list_messages_result()`. (Note: much of this was already extracted to capabilities.py in v1.3; remaining code is the glue.)

- [ ] **C-7. Defensive Chaining Without Null-Safe Helper**
  - `formatter.py:157-165`
  - Repetitive `getattr(obj, "attr", None)` chains for sender name resolution. Same pattern repeated elsewhere for media, venue, etc.
  - **Fix:** Extract `_safe_get_nested(obj, *attrs, default=None)` helper.

- [ ] **C-8. Inconsistent Input Validation Strategy**
  - `tools.py:1202-1252` (ListMessages uses `model_validator`) vs `tools.py:1361-1368` (SearchMessages relies on field requirements only)
  - **Fix:** Standardize on Pydantic validators for all ToolArgs subclasses.

- [ ] **C-9. Leaky Abstraction in Formatter: Untyped Message Parameter**
  - `formatter.py:9-15`
  - `messages: list` (untyped), `reply_map: dict[int, object]`, callbacks undocumented. Callers must read implementation to understand expected shape.
  - **Fix:** Define `MessageLike` Protocol and `TopicNameGetter` type alias. Document callback expectations.

- [x] **C-10. Dead Code: `EntityCache.all_names()`**
  - `cache.py:252-255`
  - Orphaned by `all_names_with_ttl()`. No callers.
  - **Fix:** Remove.

- [x] **C-11. Dead Imports in tools.py**
  - `tools.py:20`
  - `from telethon import TelegramClient, custom, functions, types` — unused after v1.3 capability extraction.
  - **Fix:** Remove unused imports.

## Security Analyst

**Assessment:** Read-only design is a strong security invariant. No write operations means prompt injection from message content can't trigger destructive actions. SQLite queries use parameterized statements (with one DDL exception). Session security relies on Telethon's encrypted session files.

**Risks:** Session directory created with default umask. No rate limiting. Catch-all exception handlers mask security-relevant failures.

**Open question:** Is the server ever exposed to untrusted MCP clients, or is single-user-behind-Docker the only deployment model?

### Findings

- [x] **S-1. SQL Injection via Dynamic ALTER TABLE** (Critical)
  - `cache.py:136-138`
  - `conn.execute(f"ALTER TABLE topic_metadata ADD COLUMN {column_name} {column_type}")` — f-string interpolation in DDL. Currently safe (values from hardcoded `_TOPIC_REQUIRED_COLUMNS` dict), but dangerous pattern if dict is ever populated from external input. SQLite doesn't support parameterized DDL.
  - **Fix:** Add whitelist validation: assert column_name and column_type are in allowed sets before execute.

- [ ] **S-2. Unvalidated Exception Messages in Error Responses** (High)
  - `server.py:44-58`
  - `detail = str(exc).strip()` — exception messages from Telegram RPC errors and validation errors embedded directly in user-facing responses. Could leak internal details.
  - **Mitigating factor:** Already truncated to 160 chars at `server.py:42`.
  - **Fix:** Consider returning only categorical error types for validation errors. Current truncation is acceptable for runtime errors.

- [x] **S-3. No Input Length Validation on Query Parameters** (Medium-High)
  - `tools.py:1367` (SearchMessages.query), `resolver.py:121`
  - No maximum length limits. Malicious consumer could send very large queries causing memory exhaustion, Telegram API abuse, slow fuzzy matching.
  - **Fix:** Add `Field(max_length=500)` to string params (query, dialog, sender, topic).

- [x] **S-4. Catch-All Exception Handlers Masking Errors** (High)
  - `resolver.py:108-115`
  - `except Exception: pass` silently ignores cache corruption, SQLite locking errors, and other critical failures. Makes debugging impossible.
  - **Fix:** Catch `(sqlite3.Error, OSError)` specifically. Log unexpected exceptions at WARNING level.

- [x] **S-5. Session Directory Created with Default Umask** (Medium)
  - `telegram.py:66-68`
  - `state_home.mkdir(parents=True, exist_ok=True)` — no explicit permissions. Default umask (022) makes session files world-readable. Session files contain authentication tokens.
  - **Fix:** Add `mode=0o700` to `mkdir()`.

- [ ] **S-6. No Rate Limiting on Tool Calls** (Medium)
  - `tools.py` — all tool implementations
  - No rate limiting at server or tool level. Rogue MCP client can spam calls, exceeding Telegram API rate limits and causing session bans.
  - **Panel resolution:** Defer — single-user Docker deployment. Document the assumption. Add if multi-tenant scenario appears.

- [ ] **S-7. Weak Pagination Token Validation Post-Decode** (Low-Medium)
  - `pagination.py:54-88`
  - After decoding navigation tokens, no validation that claimed dialog_id and topic_id are accessible. Attacker could craft token pointing to unauthorized dialogs.
  - **Mitigating factor:** Telegram RPC calls fail with "forbidden" error downstream.
  - **Fix:** Validate token claims match the resolved dialog/topic before using.

- [ ] **S-8. Logging May Expose User Data** (Low)
  - `tools.py:1039, 1260, 1385`
  - `logger.info("method[ListMessages] args[%s]", args)` logs full Pydantic model args including dialog names, sender names, query text.
  - **Fix:** Create safe log formatter that excludes PII-adjacent fields. Or log only tool name + non-sensitive metadata.

- [ ] **S-9. Single Shared Telegram Session** (Info — by design)
  - `telegram.py:57-68`
  - All tool calls share one authenticated session. If exposed to multiple users, all see same data.
  - **Panel resolution:** By-design single-session per deployment. Document the assumption explicitly.

## Performance / Efficiency Engineer

**Assessment:** The hot path (tool call -> capability -> Telegram RPC -> format -> response) is clean. WAL mode and TTL caching are well-designed. Navigation tokens avoid re-querying state.

**Risks:** N+1 reaction name fetches. Full entity cache loaded on every resolve(). Per-entity commit() in upsert.

**Open question:** How many entities does a typical user's cache hold? If <1000, full-load pattern is fine. If >10k, it's a problem.

### Findings

- [ ] **P-1. N+1 Reaction Name Fetches Per Message** (High)
  - `tools.py:40` (REACTION_NAMES_THRESHOLD), usage in capabilities.py
  - Reaction names fetched per message (only if reactions <= 15). For a page with 100 messages, this triggers up to 100 separate RPC calls to Telegram.
  - **Fix:** Batch reaction fetches across all messages on a page, or implement page-wide limit instead of per-message. Profile first to measure actual impact with typical page sizes (default 50).

- [ ] **P-2. Full Entity Cache Loaded on Every resolve() Call** (Medium)
  - `tools.py:261`
  - `resolve(query, cache.all_names_with_ttl(USER_TTL, GROUP_TTL), cache)` loads entire entity cache into dict. With 10k entities, allocates 10k+ dict entries per call.
  - **Panel resolution:** Profile first. If typical cache <5k entries, current approach is acceptable. Fix only if measured as bottleneck.
  - **Fix (if needed):** Implement cursor-based cache queries or in-memory name index that updates on upsert.

- [x] **P-3. Synchronous SQLite Commits in Hot Path** (Medium)
  - `cache.py:229, 342, 448, 464, 479`
  - Every entity/reaction/topic upsert calls `.commit()` immediately. ListDialogs caching 200 dialogs triggers 200 synchronous writes.
  - **Fix:** Add `upsert_batch()` method using `executemany` + single commit. Apply to ListDialogs entity warmup path.

- [ ] **P-4. Unbounded Message Buffering in Topic Reads** (Medium)
  - `tools.py:850-898` (`_fetch_topic_messages`)
  - Collects messages into list, filters by topic. Could buffer many messages before finding enough topic matches.
  - **Mitigating factor:** Loop exits when `len(raw_messages) < batch_size` (no more messages). Not truly unbounded.
  - **Fix:** Consider pushing topic filtering closer to Telegram API level if possible.

- [ ] **P-5. Repeated Topic Metadata Queries Within TTL Window** (Low-Medium)
  - `tools.py:453-492`
  - Topic cache has 10-minute TTL, but each call does full SQLite query even for same dialog. No in-memory fast path.
  - **Fix:** Add L1 in-memory cache with short TTL (30s) in front of SQLite for hot dialogs.

- [ ] **P-6. All Context Messages Pre-Loaded for Search Hits** (Low-Medium)
  - `formatter.py:98-119`
  - SearchMessages loads all context messages into `context_messages_by_id` dict before formatting. For 20 hits with context_radius=3, ~140 messages in memory.
  - **Panel resolution:** Acceptable for typical usage (20 hits x 7 context = 140 messages, ~few KB). Not a real problem.

- [ ] **P-7. Hardcoded Pagination Sizes** (Low)
  - `tools.py:41` (`FORUM_TOPICS_PAGE_SIZE = 100`)
  - For dialogs with 10k+ topics, requires 100+ round trips. This is a Telegram API constraint — page size is server-limited anyway.
  - **Panel resolution:** Acceptable. Telegram limits page size.

## Panel Conflicts & Resolutions

| Topic | Position A | Position B | Resolution |
|-------|-----------|-----------|------------|
| SQL f-string in ALTER TABLE | Security: Critical risk | Architect: Controlled dict, acceptable | **Security wins** — add whitelist validation. Low effort, eliminates risk class |
| Rate limiting | Security: Must-have | Kaizen: YAGNI for single-user Docker | **Defer** — document single-user assumption, add if multi-tenant appears |
| Full entity cache reload | Performance: Switch to index | Architect: Simpler, correct | **Defer** — profile first. If cache <5k, current approach is fine |
| Exception messages in errors | Security: Leak info | Architect: Useful for debugging | **Compromise** — already truncated to 160 chars; acceptable |
| Unbounded message buffering | Performance: OOM risk | Architect: Not truly unbounded | **Acceptable** — loop exits when batch returns fewer messages |
